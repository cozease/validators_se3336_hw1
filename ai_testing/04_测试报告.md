## 3. AI 驱动工作流发现的缺陷

所有缺陷均由 AI 生成测试首次触发，经人工复核确认为上游实现与规范/文档
之间的偏差；在测试代码中以 `@pytest.mark.xfail(strict=True, reason=…)`
方式保留并自动在日常 CI 中回归监控。

### BUG-001 · 中严重级 · `finance.isin` 校验位永不生效

- **现象**：任何 12 位字符且前 2 位为字母的字符串都会被 `isin` 判为合法。
- **复现**：
  ```python
  >>> from validators import isin
  >>> bool(isin("US0378331004"))   # 错校验位
  True
  >>> bool(isin("BADINVALID01"))   # 完全随机
  True
  ```
- **根因**：`src/validators/finance.py::_isin_checksum` 在 `for idx in range(12)`
  循环中计算出 `val` 后**未执行** `check += val // 10 + val % 10` 等累加操作，
  导致 `check` 始终等于 0，最终 `(check % 10) == 0` 恒为 True。
- **建议修复**：在循环体内追加
  `check = check + (val // 10) + (val % 10)`（与 `_cusip_checksum` 对齐）。
- **关联用例**：`test_ai_finance.py::TestISIN::test_bad_checksum_should_fail`、
  `test_random_should_fail`。

### BUG-002 · 低严重级 · `card.mastercard` 前缀范围过宽，误接 Mir 卡号

- **现象**：`mastercard('2200123456789019')` 返回 `True`，但该号段属于
  Mir 卡（`2200–2204`）。
- **根因**：`src/validators/card.py::mastercard` 中正则
  `^(51|52|53|54|55|22|23|24|25|26|27)` 对 2-series 的描述过于宽松。
  根据 Mastercard 官方规范，仅 BIN 范围 `2221–2720` 属 Mastercard，
  需要 4 位前缀级匹配（如 `^(5[1-5]\d{2}|222[1-9]|22[3-9]\d|2[3-6]\d{2}|27[01]\d|2720)`）。
- **建议修复**：把品牌正则升级到 4 位 BIN 精度。
- **关联用例**：`test_ai_card.py::TestBrandMatrix::test_mir_should_not_match_mastercard`。

### BUG-003 · 设计/文档偏差 · `_isin_checksum` 对数字-字母前缀存在矛盾

- **现象**：ISIN 国际标准允许首两位为国家代码（字母），后 9 位为本地编码
  （字母或数字），最后 1 位校验。`_isin_checksum` 使用
  `if c >= "0" and c <= "9" and idx > 1:` 接纳数字，但前两位仅接纳字母，
  与 ISIN 规范一致；然而由于 BUG-001 的累加缺失，即便格式正确也无法校验。
- **结论**：BUG-003 合并入 BUG-001 跟踪。

## 5. 结论

- 总体用例通过率 **100 %**（排除 3 个已归档的 xfail 缺陷）。
- 核心模块（email / url / card / ip / uuid / hash / encoding / between /
  length / iban）行覆盖率已达 **100 %**；总覆盖率 **92 %**，超过
  《测试计划》通过阈值。
- AI 驱动工作流在本轮迭代中定位了 **2 个真实的上游缺陷**（BUG-001、
  BUG-002），提交了最小复现用例与修复建议。
- 结论：**测试计划达成**，建议将 `ai_testing/generated_tests/` 纳入
  主仓库 CI，以便在 Mastercard 正则或 ISIN 校验修复后自动转绿。
